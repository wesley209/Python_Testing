105 class Process(metaclass=NoPublicConstructor):
106 r"""A child process. Like :class:`subprocess.Popen`, but async.
107  
108 This class has no public constructor. The most common way to get a
109 `Process` object is to combine `Nursery.start` with `run_process`::
110  
111 process_object = await nursery.start(run_process, ...)
112  
113 This way, `run_process` supervises the process and makes sure that it is
114 cleaned up properly, while optionally checking the return value, feeding
115 it input, and so on.
116  
117 If you need more control – for example, because you want to spawn a child
118 process that outlives your program – then another option is to use
119 `trio.lowlevel.open_process`::
120  
121 process_object = await trio.lowlevel.open_process(...)
122  
123 Attributes:
124 args (str or list): The ``command`` passed at construction time,
125 specifying the process to execute and its arguments.
126 pid (int): The process ID of the child process managed by this object.
127 stdin (trio.abc.SendStream or None): A stream connected to the child's
128 standard input stream: when you write bytes here, they become available
129 for the child to read. Only available if the :class:`Process`
130 was constructed using ``stdin=PIPE``; otherwise this will be None.
131 stdout (trio.abc.ReceiveStream or None): A stream connected to
132 the child's standard output stream: when the child writes to
133 standard output, the written bytes become available for you
134 to read here. Only available if the :class:`Process` was
135 constructed using ``stdout=PIPE``; otherwise this will be None.
136 stderr (trio.abc.ReceiveStream or None): A stream connected to
137 the child's standard error stream: when the child writes to
138 standard error, the written bytes become available for you
139 to read here. Only available if the :class:`Process` was
140 constructed using ``stderr=PIPE``; otherwise this will be None.
141 stdio (trio.StapledStream or None): A stream that sends data to
142 the child's standard input and receives from the child's standard
143 output. Only available if both :attr:`stdin` and :attr:`stdout` are
144 available; otherwise this will be None.

322

465 async def _run_process(
466 command: StrOrBytesPath | Sequence[StrOrBytesPath],
467 *,
468 stdin: bytes | bytearray | memoryview | int | HasFileno | None = b"",
469 capture_stdout: bool = False,
470 capture_stderr: bool = False,
471 check: bool = True,
472 deliver_cancel: Callable[[Process], Awaitable[object]] | None = None,
473 task_status: TaskStatus[Process] = trio.TASK_STATUS_IGNORED,
474 **options: object,
475 ) -> subprocess.CompletedProcess[bytes]:
476 """Run ``command`` in a subprocess and wait for it to complete.
477  
478 This function can be called in two different ways.
479  
480 One option is a direct call, like::
481  
482 completed_process_info = await trio.run_process(...)
483  
484 In this case, it returns a :class:`subprocess.CompletedProcess` instance
485 describing the results. Use this if you want to treat a process like a
486 function call.
487  
488 The other option is to run it as a task using `Nursery.start` – the enhanced version
489 of `~Nursery.start_soon` that lets a task pass back a value during startup::
490  
491 process = await nursery.start(trio.run_process, ...)
492  
493 In this case, `~Nursery.start` returns a `Process` object that you can use
494 to interact with the process while it's running. Use this if you want to
495 treat a process like a background task.
496  
497 Either way, `run_process` makes sure that the process has exited before
498 returning, handles cancellation, optionally checks for errors, and
499 provides some convenient shorthands for dealing with the child's
500 input/output.
501  
502 **Input:** `run_process` supports all the same ``stdin=`` arguments as
503 `subprocess.Popen`. In addition, if you simply want to pass in some fixed
504 data, you can pass a plain `bytes` object, and `run_process` will take
505 care of setting up a pipe, feeding in the data you gave, and then sending
506 end-of-file. The default is ``b""``, which means that the child will receive
507 an empty stdin. If you want the child to instead read from the parent's
508 stdin, use ``stdin=None``.
509  
510 **Output:** By default, any output produced by the subprocess is
511 passed through to the standard output and error streams of the
512 parent Trio process.
513  
514 When calling `run_process` directly, you can capture the subprocess's output by
515 passing ``capture_stdout=True`` to capture the subprocess's standard output, and/or
516 ``capture_stderr=True`` to capture its standard error. Captured data is collected up
517 by Trio into an in-memory buffer, and then provided as the
518 :attr:`~subprocess.CompletedProcess.stdout` and/or
519 :attr:`~subprocess.CompletedProcess.stderr` attributes of the returned
520 :class:`~subprocess.CompletedProcess` object. The value for any stream that was not
521 captured will be ``None``.
522  
523 If you want to capture both stdout and stderr while keeping them
524 separate, pass ``capture_stdout=True, capture_stderr=True``.
525  
526 If you want to capture both stdout and stderr but mixed together
527 in the order they were printed, use: ``capture_stdout=True, stderr=subprocess.STDOUT``.
528 This directs the child's stderr into its stdout, so the combined
529 output will be available in the `~subprocess.CompletedProcess.stdout`
530 attribute.
531  
532 If you're using ``await nursery.start(trio.run_process, ...)`` and want to capture
533 the subprocess's output for further processing, then use ``stdout=subprocess.PIPE``
534 and then make sure to read the data out of the `Process.stdout` stream. If you want
535 to capture stderr separately, use ``stderr=subprocess.PIPE``. If you want to capture
536 both, but mixed together in the correct order, use ``stdout=subprocess.PIPE,
537 stderr=subprocess.STDOUT``.
538  
539 **Error checking:** If the subprocess exits with a nonzero status
540 code, indicating failure, :func:`run_process` raises a
541 :exc:`subprocess.CalledProcessError` exception rather than
542 returning normally. The captured outputs are still available as
543 the :attr:`~subprocess.CalledProcessError.stdout` and
544 :attr:`~subprocess.CalledProcessError.stderr` attributes of that
545 exception. To disable this behavior, so that :func:`run_process`
546 returns normally even if the subprocess exits abnormally, pass ``check=False``.
547  
548 Note that this can make the ``capture_stdout`` and ``capture_stderr``
549 arguments useful even when starting `run_process` as a task: if you only
550 care about the output if the process fails, then you can enable capturing
551 and then read the output off of the `~subprocess.CalledProcessError`.
552  
553 **Cancellation:** If cancelled, `run_process` sends a termination
554 request to the subprocess, then waits for it to fully exit. The
555 ``deliver_cancel`` argument lets you control how the process is terminated.
556  
557 .. note:: `run_process` is intentionally similar to the standard library
558 `subprocess.run`, but some of the defaults are different. Specifically, we
559 default to:
560  
561 - ``check=True``, because `"errors should never pass silently / unless
562 explicitly silenced" <https://www.python.org/dev/peps/pep-0020/>`__.
563  
564 - ``stdin=b""``, because it produces less-confusing results if a subprocess
565 unexpectedly tries to read from stdin.
566  
567 To get the `subprocess.run` semantics, use ``check=False, stdin=None``.
568  
569 Args:
570 command (list or str): The command to run. Typically this is a
571 sequence of strings such as ``['ls', '-l', 'directory with spaces']``,
572 where the first element names the executable to invoke and the other
573 elements specify its arguments. With ``shell=True`` in the
574 ``**options``, or on Windows, ``command`` may alternatively
575 be a string, which will be parsed following platform-dependent
576 :ref:`quoting rules <subprocess-quoting>`.
577  
578 stdin (:obj:`bytes`, subprocess.PIPE, file descriptor, or None): The
579 bytes to provide to the subprocess on its standard input stream, or
580 ``None`` if the subprocess's standard input should come from the
581 same place as the parent Trio process's standard input. As is the
582 case with the :mod:`subprocess` module, you can also pass a file
583 descriptor or an object with a ``fileno()`` method, in which case
584 the subprocess's standard input will come from that file.
585  
586 When starting `run_process` as a background task, you can also use
587 ``stdin=subprocess.PIPE``, in which case `Process.stdin` will be a
588 `~trio.abc.SendStream` that you can use to send data to the child.
589  
590 capture_stdout (bool): If true, capture the bytes that the subprocess
591 writes to its standard output stream and return them in the
592 `~subprocess.CompletedProcess.stdout` attribute of the returned
593 `subprocess.CompletedProcess` or `subprocess.CalledProcessError`.
594  
595 capture_stderr (bool): If true, capture the bytes that the subprocess
596 writes to its standard error stream and return them in the
597 `~subprocess.CompletedProcess.stderr` attribute of the returned
598 `~subprocess.CompletedProcess` or `subprocess.CalledProcessError`.
599  
600 check (bool): If false, don't validate that the subprocess exits
601 successfully. You should be sure to check the
602 ``returncode`` attribute of the returned object if you pass
603 ``check=False``, so that errors don't pass silently.
604  
605 deliver_cancel (async function or None): If `run_process` is cancelled,
606 then it needs to kill the child process. There are multiple ways to
607 do this, so we let you customize it.
608  
609 If you pass None (the default), then the behavior depends on the
610 platform:
611  
612 - On Windows, Trio calls ``TerminateProcess``, which should kill the
613 process immediately.
614  
615 - On Unix-likes, the default behavior is to send a ``SIGTERM``, wait
616 5 seconds, and send a ``SIGKILL``.
617  
618 Alternatively, you can customize this behavior by passing in an
619 arbitrary async function, which will be called with the `Process`
620 object as an argument. For example, the default Unix behavior could
621 be implemented like this::
622  
623 async def my_deliver_cancel(process):
624 process.send_signal(signal.SIGTERM)
625 await trio.sleep(5)
626 process.send_signal(signal.SIGKILL)
627  
628 When the process actually exits, the ``deliver_cancel`` function
629 will automatically be cancelled – so if the process exits after
630 ``SIGTERM``, then we'll never reach the ``SIGKILL``.

883 """Run ``command`` in a subprocess and wait for it to complete.
884  
885 This function can be called in two different ways.
886  
887 One option is a direct call, like::
888  
889 completed_process_info = await trio.run_process(...)
890  
891 In this case, it returns a :class:`subprocess.CompletedProcess` instance
892 describing the results. Use this if you want to treat a process like a
893 function call.
894  
895 The other option is to run it as a task using `Nursery.start` – the enhanced version
896 of `~Nursery.start_soon` that lets a task pass back a value during startup::
897  
898 process = await nursery.start(trio.run_process, ...)
899  
900 In this case, `~Nursery.start` returns a `Process` object that you can use
901 to interact with the process while it's running. Use this if you want to
902 treat a process like a background task.
903  
904 Either way, `run_process` makes sure that the process has exited before
905 returning, handles cancellation, optionally checks for errors, and
906 provides some convenient shorthands for dealing with the child's
907 input/output.
908  
909 **Input:** `run_process` supports all the same ``stdin=`` arguments as
910 `subprocess.Popen`. In addition, if you simply want to pass in some fixed
911 data, you can pass a plain `bytes` object, and `run_process` will take
912 care of setting up a pipe, feeding in the data you gave, and then sending
913 end-of-file. The default is ``b""``, which means that the child will receive
914 an empty stdin. If you want the child to instead read from the parent's
915 stdin, use ``stdin=None``.
916  
917 **Output:** By default, any output produced by the subprocess is
918 passed through to the standard output and error streams of the
919 parent Trio process.
920  
921 When calling `run_process` directly, you can capture the subprocess's output by
922 passing ``capture_stdout=True`` to capture the subprocess's standard output, and/or
923 ``capture_stderr=True`` to capture its standard error. Captured data is collected up
924 by Trio into an in-memory buffer, and then provided as the
925 :attr:`~subprocess.CompletedProcess.stdout` and/or
926 :attr:`~subprocess.CompletedProcess.stderr` attributes of the returned
927 :class:`~subprocess.CompletedProcess` object. The value for any stream that was not

931 separate, pass ``capture_stdout=True, capture_stderr=True``.
932

934 in the order they were printed, use: ``capture_stdout=True, stderr=subprocess.STDOUT``.
935 This directs the child's stderr into its stdout, so the combined
936 output will be available in the `~subprocess.CompletedProcess.stdout`
937 attribute.
938  
939 If you're using ``await nursery.start(trio.run_process, ...)`` and want to capture
940 the subprocess's output for further processing, then use ``stdout=subprocess.PIPE``
941 and then make sure to read the data out of the `Process.stdout` stream. If you want
942 to capture stderr separately, use ``stderr=subprocess.PIPE``. If you want to capture
943 both, but mixed together in the correct order, use ``stdout=subprocess.PIPE,
944 stderr=subprocess.STDOUT``.
945

947 code, indicating failure, :func:`run_process` raises a
948 :exc:`subprocess.CalledProcessError` exception rather than
949 returning normally. The captured outputs are still available as
950 the :attr:`~subprocess.CalledProcessError.stdout` and
951 :attr:`~subprocess.CalledProcessError.stderr` attributes of that

953 returns normally even if the subprocess exits abnormally, pass ``check=False``.
954  
955 Note that this can make the ``capture_stdout`` and ``capture_stderr``
956 arguments useful even when starting `run_process` as a task: if you only
957 care about the output if the process fails, then you can enable capturing
958 and then read the output off of the `~subprocess.CalledProcessError`.
959  
960 **Cancellation:** If cancelled, `run_process` sends a termination
961 request to the subprocess, then waits for it to fully exit. The
962 ``deliver_cancel`` argument lets you control how the process is terminated.
963  
964 .. note:: `run_process` is intentionally similar to the standard library
965 `subprocess.run`, but some of the defaults are different. Specifically, we
966 default to:
967  
968 - ``check=True``, because `"errors should never pass silently / unless
969 explicitly silenced" <https://www.python.org/dev/peps/pep-0020/>`__.
970  
971 - ``stdin=b""``, because it produces less-confusing results if a subprocess

973
