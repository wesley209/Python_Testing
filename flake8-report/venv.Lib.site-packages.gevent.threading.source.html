101 # We avoid calling the superclass constructor. This makes us about
102 # twice as fast:
103 #
104 # - 1.16 vs 0.68usec on PyPy (unknown version, older Intel mac)
105 # - 29.3 vs 17.7usec on CPython 2.7 (older intel Mac)
106 # - 0.98 vs 2.95usec on CPython 3.12.2 (newer M2 mac)
107 #
108 # It als has the important effect of avoiding allocation and then
109 # immediate deletion of _Thread__block, a lock. This is especially
110 # important on PyPy where locks go through the cpyext API and
111 # Cython, which is known to be slow and potentially buggy (e.g.,
112 # https://bitbucket.org/pypy/pypy/issues/2149/memory-leak-for-python-subclass-of-cpyext#comment-22347393)

114 # These objects are constructed quite frequently in some cases, so
115 # the optimization matters: for example, in gunicorn, which uses
116 # pywsgi.WSGIServer, most every request is handled in a new greenlet,
117 # and every request uses a logging.Logger to write the access log,
118 # and every call to a log method captures the current thread (by
119 # default).
120 #
121 # (Obviously we have to duplicate the effects of the constructor,
122 # at least for external state purposes, which is potentially
123 # slightly fragile.)

126 # in ``threading._active``

128 # This class also solves a problem forking process with subprocess: after forking,

190 # In Python 3.11.8+ and 3.12.2+ (yes, minor patch releases),
191 # CPython's ``threading._after_fork`` hook began swizzling the
192 # type of the _DummyThread into _MainThread if such a dummy thread
193 # was the current thread when ``os.fork()`` gets called.
194 # From CPython's perspective, that's a more-or-less fine thing to do.
195 # While _DummyThread isn't a subclass of _MainThread, they are both
196 # subclasses of Thread, and _MainThread doesn't add any new instance
197 # variables.
198 #
199 # From gevent's perspective, that's NOT good. Our _DummyThread
200 # doesn't have all the instance variables that Thread does, and so
201 # attempting to do anything with this now-fake _MainThread doesn't work.
202 # You in fact immediately get assertion errors from inside ``_after_fork``.
203 # Now, these are basically harmless --- they're printed, and they prevent the cleanup
204 # of some globals in _threading, but that probably doesn't matter --- but
205 # people complained, and it could break some test scenarios (due to unexpected
206 # output on stderr, for example)
207 #
208 # We thought of a few options to patch around this:
209 #
210 # - Live with the performance penalty. Newer CPythons are making it
211 # harder and harder to perform well, so if we can possibly avoid
212 # adding our own performance regressions, that would be good.
213 #
214 # - ``after_fork`` uses ``isinstance(current, _DummyThread)``
215 # before swizzling, so we could use a metaclass to make that
216 # check return false. That's a fairly large compatibility risk,
217 # both because of the use of a metaclass (what if some other
218 # subclass of _DummyTHread is using an incompatible metaclass?)
219 # and the change in ``isinstance`` behaviour. We could limit the latter
220 # to a window around the fork, using ``os.register_at_fork(before, after_in_parent=)``,
221 # but that's a lot of moving pieces requiring the use of a global or class
222 # variable to track state.
223 #
224 # - We could copy the ivars of the current main thread into the
225 # _DummyThread in ``register_at_fork(before=)``. That appears to
226 # work, but also requires the use of
227 # ``register_at_fork(after_in_parent=)`` to reverse it.
228 #
229 # - We could simply prevent swizzling the class in the first
230 # place. In combination with
231 # ``register_at_fork(after_in_child=)`` to establish a *real*
232 # new _MainThread, that's a clean solution. Establishing a real
233 # new _MainThread is something that CPython itself is prepared
234 # to do if it can't figure out what the current thread is. The
235 # compatibility risk of this is relatively low: swizzling
236 # classes is frowned upon and uncommon, and we can limit it to
237 # just preventing this specific case. And if somebody was
238 # attempting this already with some other thread subclass, it
239 # would (probably?) have the exact same issues, so we can be pretty
240 # sure nobody is doing that.
241 #
242 # We're initially going with the last fix; the __class__ part is here,
243 # the ``after_in_child`` fixup we only apply if we're monkey-patching.
244 #
245 # Now, all of this is moot in 3.13, which takes a very different
246 # approach to handling this, and also changes some names. See
247 # https://github.com/python/cpython/commit/0e9c364f4ac18a2237bdbac702b96bcf8ef9cb09
248  
249 # Tests pass just fine in 3.8 (and presumably 3.9 and 3.10) with these fixes
250 # applied, but just in case, we only do it where we know it's necessary.
251 _NEEDS_CLASS_FORK_FIXUP = (
252 (sys.version_info[:2] == (3, 11) and sys.version_info[:3] >= (3, 11, 8))
253 or sys.version_info[:3] >= (3, 12, 2)
254 )
