2 """
3 Tests for greenlets interacting with the CPython trash can API.
4  
5 The CPython trash can API is not designed to be re-entered from a
6 single thread. But this can happen using greenlets, if something
7 during the object deallocation process switches greenlets, and this second
8 greenlet then causes the trash can to get entered again. Here, we do this
9 very explicitly, but in other cases (like gevent) it could be arbitrarily more
10 complicated: for example, a weakref callback might try to acquire a lock that's
11 already held by another greenlet; that would allow a greenlet switch to occur.
12  
13 See https://github.com/gevent/gevent/issues/1909
14  
15 This test is fragile and relies on details of the CPython
16 implementation (like most of the rest of this package):
17  
18 - We enter the trashcan and deferred deallocation after
19 ``_PyTrash_UNWIND_LEVEL`` calls. This constant, defined in
20 CPython's object.c, is generally 50. That's basically how many objects are required to
21 get us into the deferred deallocation situation.
22  
23 - The test fails by hitting an ``assert()`` in object.c; if the
24 build didn't enable assert, then we don't catch this.
25  
26 - If the test fails in that way, the interpreter crashes.
27 """
