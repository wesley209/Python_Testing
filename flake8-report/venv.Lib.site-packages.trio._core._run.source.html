264 def _prune(self) -> None:

266 # forth repeatedly between the same two deadlines, and end up with
267 # lots of stale entries that *look* like they're still active, because

280 # this assert:

303 class CancelStatus:
304 """Tracks the cancellation status for a contiguous extent
305 of code that will become cancelled, or not, as a unit.
306  
307 Each task has at all times a single "active" CancelStatus whose
308 cancellation state determines whether checkpoints executed in that
309 task raise Cancelled. Each 'with CancelScope(...)' context is
310 associated with a particular CancelStatus. When a task enters
311 such a context, a CancelStatus is created which becomes the active
312 CancelStatus for that task; when the 'with' block is exited, the
313 active CancelStatus for that task goes back to whatever it was
314 before.
315  
316 CancelStatus objects are arranged in a tree whose structure
317 mirrors the lexical nesting of the cancel scope contexts. When a
318 CancelStatus becomes cancelled, it notifies all of its direct
319 children, who become cancelled in turn (and continue propagating
320 the cancellation down the tree) unless they are shielded. (There
321 will be at most one such child except in the case of a
322 CancelStatus that immediately encloses a nursery.) At the leaves
323 of this tree are the tasks themselves, which get woken up to deliver
324 an abort when their direct parent CancelStatus becomes cancelled.
325  
326 You can think of CancelStatus as being responsible for the
327 "plumbing" of cancellations as oppposed to CancelScope which is
328 responsible for the origination of them.

445 # statuses, to ensure their state is up-to-date. It's basically a
446 # recursive algorithm, but we use an explicit stack to avoid any
447 # issues with stack overflow.

501 class CancelScope:
502 """A *cancellation scope*: the link between a unit of cancellable
503 work and Trio's cancellation system.
504  
505 A :class:`CancelScope` becomes associated with some cancellable work
506 when it is used as a context manager surrounding that work::
507  
508 cancel_scope = trio.CancelScope()
509 ...
510 with cancel_scope:
511 await long_running_operation()
512  
513 Inside the ``with`` block, a cancellation of ``cancel_scope`` (via
514 a call to its :meth:`cancel` method or via the expiry of its
515 :attr:`deadline`) will immediately interrupt the
516 ``long_running_operation()`` by raising :exc:`Cancelled` at its
517 next :ref:`checkpoint <checkpoints>`.
518  
519 The context manager ``__enter__`` returns the :class:`CancelScope`
520 object itself, so you can also write ``with trio.CancelScope() as
521 cancel_scope:``.
522  
523 If a cancel scope becomes cancelled before entering its ``with`` block,
524 the :exc:`Cancelled` exception will be raised at the first
525 checkpoint inside the ``with`` block. This allows a
526 :class:`CancelScope` to be created in one :ref:`task <tasks>` and
527 passed to another, so that the first task can later cancel some work
528 inside the second.
529  
530 Cancel scopes are not reusable or reentrant; that is, each cancel
531 scope can be used for at most one ``with`` block. (You'll get a
532 :exc:`RuntimeError` if you violate this rule.)
533  
534 The :class:`CancelScope` constructor takes initial values for the
535 cancel scope's :attr:`deadline` and :attr:`shield` attributes; these
536 may be freely modified after construction, whether or not the scope
537 has been entered yet, and changes take immediate effect.

1294 and it must make sure that it (or someone) eventually calls
1295 :meth:`task_status.started() <TaskStatus.started>`.

1298

1305 a do-nothing ``started`` method. This way your function supports

1307 arg2)`` or directly like ``await async_fn(arg1, arg2)``, and
1308 either way it can call :meth:`task_status.started() <TaskStatus.started>`
1309 without worrying about which mode it's in. Defining your function like
1310 this will make it obvious to readers that it supports being used
1311 in both modes.
1312  
1313 Before the child calls :meth:`task_status.started() <TaskStatus.started>`,
1314 it's effectively run underneath the call to :meth:`start`: if it
1315 raises an exception then that exception is reported by
1316 :meth:`start`, and does *not* propagate out of the nursery. If
1317 :meth:`start` is cancelled, then the child task is also
1318 cancelled.
1319  
1320 When the child calls :meth:`task_status.started() <TaskStatus.started>`,
1321 it's moved out from underneath :meth:`start` and into the given nursery.
1322  
1323 If the child task passes a value to :meth:`task_status.started(value) <TaskStatus.started>`,
1324 then :meth:`start` returns this value. Otherwise, it returns ``None``.

1331 # to avoid wrapping pre-started() exceptions in an extra ExceptionGroup.

1334 # set strict_exception_groups = True to make sure we always unwrap
1335 # *this* nursery's exceptiongroup
1336 async with open_nursery(strict_exception_groups=True) as old_nursery:

1338 thunk = functools.partial(async_fn, task_status=task_status)

1342 old_nursery,

1347 # cancel this nursery:
1348 except BaseExceptionGroup as exc:

2184 # run
2185 ################################################################
2186 #
2187 # Trio's core task scheduler and coroutine runner is in 'unrolled_run'. It's
2188 # called that because it has an unusual feature: it's actually a generator.
2189 # Whenever it needs to fetch IO events from the OS, it yields, and waits for
2190 # its caller to send the IO events back in. So the loop is "unrolled" into a
2191 # sequence of generator send() calls.
2192 #
2193 # The reason for this unusual design is to support two different modes of
2194 # operation, where the IO is handled differently.
2195 #
2196 # In normal mode using trio.run, the scheduler and IO run in the same thread:
2197 #
2198 # Main thread:
2199 #
2200 # +---------------------------+
2201 # | Run tasks |
2202 # | (unrolled_run) |
2203 # +---------------------------+
2204 # | Block waiting for I/O |
2205 # | (io_manager.get_events) |
2206 # +---------------------------+
2207 # | Run tasks |
2208 # | (unrolled_run) |
2209 # +---------------------------+
2210 # | Block waiting for I/O |
2211 # | (io_manager.get_events) |
2212 # +---------------------------+
2213 # :
2214 #
2215 #
2216 # In guest mode using trio.lowlevel.start_guest_run, the scheduler runs on the
2217 # main thread as a host loop callback, but blocking for IO gets pushed into a
2218 # worker thread:
2219 #
2220 # Main thread executing host loop: Trio I/O thread:
2221 #
2222 # +---------------------------+
2223 # | Run Trio tasks |
2224 # | (unrolled_run) |
2225 # +---------------------------+ --------------+
2226 # v
2227 # +---------------------------+ +----------------------------+
2228 # | Host loop does whatever | | Block waiting for Trio I/O |
2229 # | it wants | | (io_manager.get_events) |
2230 # +---------------------------+ +----------------------------+
2231 # |
2232 # +---------------------------+ <-------------+
2233 # | Run Trio tasks |
2234 # | (unrolled_run) |
2235 # +---------------------------+ --------------+
2236 # v
2237 # +---------------------------+ +----------------------------+
2238 # | Host loop does whatever | | Block waiting for Trio I/O |
2239 # | it wants | | (io_manager.get_events) |
2240 # +---------------------------+ +----------------------------+
2241 # : :
2242 #
2243 # Most of Trio's internals don't need to care about this difference. The main
2244 # complication it creates is that in guest mode, we might need to wake up not
2245 # just due to OS-reported IO events, but also because of code running on the
2246 # host loop calling reschedule() or changing task deadlines. Search for
