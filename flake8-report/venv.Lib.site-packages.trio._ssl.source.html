19 # General theory of operation:
20 #
21 # We implement an API that closely mirrors the stdlib ssl module's blocking
22 # API, and we do it using the stdlib ssl module's non-blocking in-memory API.
23 # The stdlib non-blocking in-memory API is barely documented, and acts as a
24 # thin wrapper around openssl, whose documentation also leaves something to be
25 # desired. So here's the main things you need to know to understand the code
26 # in this file:
27 #
28 # We use an ssl.SSLObject, which exposes the four main I/O operations:
29 #
30 # - do_handshake: performs the initial handshake. Must be called once at the
31 # beginning of each connection; is a no-op once it's completed once.
32 #
33 # - write: takes some unencrypted data and attempts to send it to the remote
34 # peer.
35  
36 # - read: attempts to decrypt and return some data from the remote peer.
37 #
38 # - unwrap: this is weirdly named; maybe it helps to realize that the thing it
39 # wraps is called SSL_shutdown. It sends a cryptographically signed message
40 # saying "I'm closing this connection now", and then waits to receive the
41 # same from the remote peer (unless we already received one, in which case
42 # it returns immediately).
43 #
44 # All of these operations read and write from some in-memory buffers called
45 # "BIOs", which are an opaque OpenSSL-specific object that's basically
46 # semantically equivalent to a Python bytearray. When they want to send some
47 # bytes to the remote peer, they append them to the outgoing BIO, and when
48 # they want to receive some bytes from the remote peer, they try to pull them
49 # out of the incoming BIO. "Sending" always succeeds, because the outgoing BIO
50 # can always be extended to hold more data. "Receiving" acts sort of like a
51 # non-blocking socket: it might manage to get some data immediately, or it
52 # might fail and need to be tried again later. We can also directly add or
53 # remove data from the BIOs whenever we want.
54 #
55 # Now the problem is that while these I/O operations are opaque atomic
56 # operations from the point of view of us calling them, under the hood they
57 # might require some arbitrary sequence of sends and receives from the remote
58 # peer. This is particularly true for do_handshake, which generally requires a
59 # few round trips, but it's also true for write and read, due to an evil thing
60 # called "renegotiation".
61 #
62 # Renegotiation is the process by which one of the peers might arbitrarily
63 # decide to redo the handshake at any time. Did I mention it's evil? It's
64 # pretty evil, and almost universally hated. The HTTP/2 spec forbids the use
65 # of TLS renegotiation for HTTP/2 connections. TLS 1.3 removes it from the
66 # protocol entirely. It's impossible to trigger a renegotiation if using
67 # Python's ssl module. OpenSSL's renegotiation support is pretty buggy [1].
68 # Nonetheless, it does get used in real life, mostly in two cases:
69 #
70 # 1) Normally in TLS 1.2 and below, when the client side of a connection wants
71 # to present a certificate to prove their identity, that certificate gets sent
72 # in plaintext. This is bad, because it means that anyone eavesdropping can
73 # see who's connecting – it's like sending your username in plain text. Not as
74 # bad as sending your password in plain text, but still, pretty bad. However,
75 # renegotiations *are* encrypted. So as a workaround, it's not uncommon for
76 # systems that want to use client certificates to first do an anonymous
77 # handshake, and then to turn around and do a second handshake (=
78 # renegotiation) and this time ask for a client cert. Or sometimes this is
79 # done on a case-by-case basis, e.g. a web server might accept a connection,
80 # read the request, and then once it sees the page you're asking for it might
81 # stop and ask you for a certificate.
82 #
83 # 2) In principle the same TLS connection can be used for an arbitrarily long
84 # time, and might transmit arbitrarily large amounts of data. But this creates
85 # a cryptographic problem: an attacker who has access to arbitrarily large
86 # amounts of data that's all encrypted using the same key may eventually be
87 # able to use this to figure out the key. Is this a real practical problem? I
88 # have no idea, I'm not a cryptographer. In any case, some people worry that
89 # it's a problem, so their TLS libraries are designed to automatically trigger
90 # a renegotiation every once in a while on some sort of timer.
91 #
92 # The end result is that you might be going along, minding your own business,
93 # and then *bam*! a wild renegotiation appears! And you just have to cope.
94 #
95 # The reason that coping with renegotiations is difficult is that some
96 # unassuming "read" or "write" call might find itself unable to progress until
97 # it does a handshake, which remember is a process with multiple round
98 # trips. So read might have to send data, and write might have to receive
99 # data, and this might happen multiple times. And some of those attempts might
100 # fail because there isn't any data yet, and need to be retried. Managing all
101 # this is pretty complicated.
102 #
103 # Here's how openssl (and thus the stdlib ssl module) handle this. All of the
104 # I/O operations above follow the same rules. When you call one of them:
105 #
106 # - it might write some data to the outgoing BIO
107 # - it might read some data from the incoming BIO
108 # - it might raise SSLWantReadError if it can't complete without reading more
109 # data from the incoming BIO. This is important: the "read" in ReadError
110 # refers to reading from the *underlying* stream.
111 # - (and in principle it might raise SSLWantWriteError too, but that never
112 # happens when using memory BIOs, so never mind)
113 #
114 # If it doesn't raise an error, then the operation completed successfully
115 # (though we still need to take any outgoing data out of the memory buffer and
116 # put it onto the wire). If it *does* raise an error, then we need to retry
117 # *exactly that method call* later – in particular, if a 'write' failed, we
118 # need to try again later *with the same data*, because openssl might have
119 # already committed some of the initial parts of our data to its output even
120 # though it didn't tell us that, and has remembered that the next time we call
121 # write it needs to skip the first 1024 bytes or whatever it is. (Well,
122 # technically, we're actually allowed to call 'write' again with a data buffer
123 # which is the same as our old one PLUS some extra stuff added onto the end,
124 # but in Trio that never comes up so never mind.)
125 #
126 # There are some people online who claim that once you've gotten a Want*Error
127 # then the *very next call* you make to openssl *must* be the same as the
128 # previous one. I'm pretty sure those people are wrong. In particular, it's
129 # okay to call write, get a WantReadError, and then call read a few times;
130 # it's just that *the next time you call write*, it has to be with the same
131 # data.
132 #
133 # One final wrinkle: we want our SSLStream to support full-duplex operation,
134 # i.e. it should be possible for one task to be calling send_all while another
135 # task is calling receive_some. But renegotiation makes this a big hassle, because
136 # even if SSLStream's restricts themselves to one task calling send_all and one
137 # task calling receive_some, those two tasks might end up both wanting to call
138 # send_all, or both to call receive_some at the same time *on the underlying
139 # stream*. So we have to do some careful locking to hide this problem from our
140 # users.
141 #
142 # (Renegotiation is evil.)
143 #
144 # So our basic strategy is to define a single helper method called "_retry",
145 # which has generic logic for dealing with SSLWantReadError, pushing data from
146 # the outgoing BIO to the wire, reading data from the wire to the incoming
147 # BIO, retrying an I/O call until it works, and synchronizing with other tasks
148 # that might be calling _retry concurrently. Basically it takes an SSLObject
149 # non-blocking in-memory method and converts it into a Trio async blocking
150 # method. _retry is only about 30 lines of code, but all these cases
151 # multiplied by concurrent calls make it extremely tricky, so there are lots
152 # of comments down below on the details, and a really extensive test suite in
153 # test_ssl.py. And now you know *why* it's so tricky, and can probably
154 # understand how it works.
155 #
156 # [1] https://rt.openssl.org/Ticket/Display.html?id=3712

177 # Ideally, when the user calls SSLStream.receive_some() with no argument, then
178 # we should do exactly one call to self.transport_stream.receive_some(),
179 # decrypt everything we got, and return it. Unfortunately, the way openssl's
180 # API works, we have to pick how much data we want to allow when we call
181 # read(), and then it (potentially) triggers a call to
182 # transport_stream.receive_some(). So at the time we pick the amount of data
183 # to decrypt, we don't know how much data we've read. As a simple heuristic,
184 # we record the max amount of data returned by previous calls to
185 # transport_stream.receive_some(), and we use that for future calls to read().
186 # But what do we use for the very first call? That's what this constant sets.
187 #
188 # Note that the value passed to read() is a limit on the amount of
189 # *decrypted* data, but we can only see the size of the *encrypted* data
190 # returned by transport_stream.receive_some(). TLS adds a small amount of
191 # framing overhead, and TLS compression is rarely used these days because it's
192 # insecure. So the size of the encrypted data should be a slight over-estimate
193 # of the size of the decrypted data, which is exactly what we want.
194 #
195 # The specific value is not really based on anything; it might be worth tuning
196 # at some point. But, if you have an TCP connection with the typical 1500 byte
197 # MTU and an initial window of 10 (see RFC 6928), then the initial burst of
198 # data will be limited to ~15000 bytes (or a bit less due to IP-level framing
199 # overhead), so this is chosen to be larger than that.
200 STARTING_RECEIVE_SIZE: TFinal = 16384

249 @final
250 class SSLStream(Stream, Generic[T_Stream]):
251 r"""Encrypted communication using SSL/TLS.
252  
253 :class:`SSLStream` wraps an arbitrary :class:`~trio.abc.Stream`, and
254 allows you to perform encrypted communication over it using the usual
255 :class:`~trio.abc.Stream` interface. You pass regular data to
256 :meth:`send_all`, then it encrypts it and sends the encrypted data on the
257 underlying :class:`~trio.abc.Stream`; :meth:`receive_some` takes encrypted
258 data out of the underlying :class:`~trio.abc.Stream` and decrypts it
259 before returning it.
260  
261 You should read the standard library's :mod:`ssl` documentation carefully
262 before attempting to use this class, and probably other general
263 documentation on SSL/TLS as well. SSL/TLS is subtle and quick to
264 anger. Really. I'm not kidding.
265  
266 Args:
267 transport_stream (~trio.abc.Stream): The stream used to transport
268 encrypted data. Required.
269  
270 ssl_context (~ssl.SSLContext): The :class:`~ssl.SSLContext` used for
271 this connection. Required. Usually created by calling
272 :func:`ssl.create_default_context`.
273  
274 server_hostname (str, bytes, or None): The name of the server being
275 connected to. Used for `SNI
276 <https://en.wikipedia.org/wiki/Server_Name_Indication>`__ and for
277 validating the server's certificate (if hostname checking is
278 enabled). This is effectively mandatory for clients, and actually
279 mandatory if ``ssl_context.check_hostname`` is ``True``.
280  
281 server_side (bool): Whether this stream is acting as a client or
282 server. Defaults to False, i.e. client mode.
283  
284 https_compatible (bool): There are two versions of SSL/TLS commonly
285 encountered in the wild: the standard version, and the version used
286 for HTTPS (HTTP-over-SSL/TLS).
287  
288 Standard-compliant SSL/TLS implementations always send a
289 cryptographically signed ``close_notify`` message before closing the
290 connection. This is important because if the underlying transport
291 were simply closed, then there wouldn't be any way for the other
292 side to know whether the connection was intentionally closed by the
293 peer that they negotiated a cryptographic connection to, or by some
294 `man-in-the-middle
295 <https://en.wikipedia.org/wiki/Man-in-the-middle_attack>`__ attacker
296 who can't manipulate the cryptographic stream, but can manipulate
297 the transport layer (a so-called "truncation attack").
298  
299 However, this part of the standard is widely ignored by real-world
300 HTTPS implementations, which means that if you want to interoperate
301 with them, then you NEED to ignore it too.
302  
303 Fortunately this isn't as bad as it sounds, because the HTTP
304 protocol already includes its own equivalent of ``close_notify``, so
305 doing this again at the SSL/TLS level is redundant. But not all
306 protocols do! Therefore, by default Trio implements the safer
307 standard-compliant version (``https_compatible=False``). But if
308 you're speaking HTTPS or some other protocol where
309 ``close_notify``\s are commonly skipped, then you should set
310 ``https_compatible=True``; with this setting, Trio will neither
311 expect nor send ``close_notify`` messages.
312  
313 If you have code that was written to use :class:`ssl.SSLSocket` and
314 now you're porting it to Trio, then it may be useful to know that a
315 difference between :class:`SSLStream` and :class:`ssl.SSLSocket` is
316 that :class:`~ssl.SSLSocket` implements the
317 ``https_compatible=True`` behavior by default.
318  
319 Attributes:
320 transport_stream (trio.abc.Stream): The underlying transport stream
321 that was passed to ``__init__``. An example of when this would be
322 useful is if you're using :class:`SSLStream` over a
323 :class:`~trio.SocketStream` and want to call the
324 :class:`~trio.SocketStream`'s :meth:`~trio.SocketStream.setsockopt`
325 method.
326  
327 Internally, this class is implemented using an instance of
328 :class:`ssl.SSLObject`, and all of :class:`~ssl.SSLObject`'s methods and
329 attributes are re-exported as methods and attributes on this class.
330 However, there is one difference: :class:`~ssl.SSLObject` has several
331 methods that return information about the encrypted connection, like
332 :meth:`~ssl.SSLSocket.cipher` or
333 :meth:`~ssl.SSLSocket.selected_alpn_protocol`. If you call them before the
334 handshake, when they can't possibly return useful data, then
335 :class:`ssl.SSLObject` returns None, but :class:`trio.SSLStream`
336 raises :exc:`NeedHandshakeError`.
337  
338 This also means that if you register a SNI callback using
339 `~ssl.SSLContext.sni_callback`, then the first argument your callback
340 receives will be a :class:`ssl.SSLObject`.
341  
342 """

459 # WARNING: this code needs to be very careful with when it
460 # calls 'await'! There might be multiple tasks calling this
461 # function at the same time trying to do different operations,
462 # so we need to be careful to:
463 #
464 # 1) interact with the SSLObject, then
465 # 2) await on exactly one thing that lets us make forward
466 # progress, then
467 # 3) loop or exit
468 #
469 # In particular we don't want to yield while interacting with
470 # the SSLObject (because it's shared state, so someone else
471 # might come in and mess with it while we're suspended), and
472 # we don't want to yield *before* starting the operation that
473 # will help us make progress, because then someone else might
474 # come in and leapfrog us.

477 #
478 # NB: despite what the docs say, SSLWantWriteError can't
479 # happen – "Writes to memory BIOs will always succeed if

481 # indefinitely."

484 ret = None

486 ret = fn(*args)

489 except (_stdlib_ssl.SSLError, _stdlib_ssl.CertificateError) as exc:

491 raise trio.BrokenResourceError from exc

499 # Some versions of SSL_do_handshake have a bug in how they handle
500 # the TLS 1.3 handshake on the server side: after the handshake
501 # finishes, they automatically send session tickets, even though
502 # the client may not be expecting data to arrive at this point and
503 # sending it could cause a deadlock or lost data. This applies at
504 # least to OpenSSL 1.1.1c and earlier, and the OpenSSL devs
505 # currently have no plans to fix it:
506 #
507 # https://github.com/openssl/openssl/issues/7948
508 # https://github.com/openssl/openssl/issues/7967
509 #
510 # The correct behavior is to wait to send session tickets on the
511 # first call to SSL_write. (This is what BoringSSL does.) So, we
512 # use a heuristic to detect when OpenSSL has tried to send session
513 # tickets, and we manually delay sending them until the
514 # appropriate moment. For more discussion see:
515 #
516 # https://github.com/python-trio/trio/issues/819#issuecomment-517529763

519 and not want_read

521 and self._ssl_object.version() == "TLSv1.3"

523 assert self._delayed_outgoing is None

527 # Outputs from the above code block are:
528 #
529 # - to_send: bytestring; if non-empty then we need to send
530 # this data to make forward progress
531 #
532 # - want_read: True if we need to receive_some some data to make
533 # forward progress
534 #
535 # - finished: False means that we need to retry the call to
536 # fn(*args) again, after having pushed things forward. True
537 # means we still need to do whatever was said (in particular
538 # send any data in to_send), but once we do then we're
539 # done.
540 #
541 # - ret: the operation's return value. (Meaningless unless
542 # finished is True.)
543 #
544 # Invariant: want_read and finished can't both be True at the
545 # same time.
546 #
547 # Now we need to move things forward. There are two things we
548 # might have to do, and any given operation might require
549 # either, both, or neither to proceed:
550 #
551 # - send the data in to_send
552 #
553 # - receive_some some data and put it into the incoming BIO
554 #
555 # Our strategy is: if there's data to send, send it;
556 # *otherwise* if there's data to receive_some, receive_some it.
557 #
558 # If both need to happen, then we only send. Why? Well, we
559 # know that *right now* we have to both send and receive_some
560 # before the operation can complete. But as soon as we yield,
561 # that information becomes potentially stale – e.g. while
562 # we're sending, some other task might go and receive_some the
563 # data we need and put it into the incoming BIO. And if it
564 # does, then we *definitely don't* want to do a receive_some –
565 # there might not be any more data coming, and we'd deadlock!
566 # We could do something tricky to keep track of whether a
567 # receive_some happens while we're sending, but the case where
568 # we have to do both is very unusual (only during a
569 # renegotiation), so it's better to keep things simple. So we
570 # do just one potentially-blocking operation, then check again
571 # for fresh information.
572 #
573 # And we prioritize sending over receiving because, if there
574 # are multiple tasks that want to receive_some, then it
575 # doesn't matter what order they go in. But if there are
576 # multiple tasks that want to send, then they each have
577 # different data, and the data needs to get put onto the wire
578 # in the same order that it was retrieved from the outgoing
579 # BIO. So if we have data to send, that *needs* to be the
580 # *very* *next* *thing* we do, to make sure no-one else sneaks
581 # in before us. Or if we can't send immediately because
582 # someone else is, then we at least need to get in line
583 # immediately.

587 yielded = True

589 if self._delayed_outgoing is not None:
590 to_send = self._delayed_outgoing + to_send
591 self._delayed_outgoing = None
592 await self.transport_stream.send_all(to_send)

595 # don't know how much. This stream is doomed.

597 raise

599 # It's possible that someone else is already blocked in
